# FitFindr

A conversational thrift-shopping agent that takes a natural language query, finds matching secondhand listings, and returns a styled outfit suggestion plus a social-media-ready fit card.

---

## Tool Inventory

### `search_listings`

**Purpose:** Finds secondhand listings that match the user's query by scoring each listing on keyword overlap, then filters by size and price ceiling.

**Inputs:**
- `description` (`str`) — keywords describing what the user wants (e.g., `"vintage graphic tee"`)
- `size` (`Optional[str]`) — size string to filter by; matching is case-insensitive; `None` skips size filtering
- `max_price` (`Optional[float]`) — inclusive price ceiling; `None` skips price filtering

**Output:** `list[dict]` — matching listing dicts sorted by relevance score, highest first. Each dict contains `id`, `title`, `description`, `category`, `style_tags`, `size`, `condition`, `price`, `colors`, `brand`, and `platform`. Returns an empty list when nothing matches; never raises.

---

### `suggest_outfit`

**Purpose:** Given a thrifted item and the user's wardrobe, asks an LLM to suggest one or two specific outfit combinations using named wardrobe pieces, or falls back to general styling advice if the wardrobe is empty.

**Inputs:**
- `new_item` (`dict`) — a listing dict for the item the user is considering buying
- `wardrobe` (`dict`) — a wardrobe dict with an `"items"` key containing a list of wardrobe item dicts; may be empty

**Output:** `str` — a non-empty outfit suggestion or general styling advice generated by the LLM. Never returns an empty string; never raises.

---

### `create_fit_card`

**Purpose:** Takes the outfit suggestion and listing details and generates a short, casual Instagram/TikTok-style caption that naturally mentions the item name, price, and platform.

**Inputs:**
- `outfit` (`str`) — the outfit suggestion string from `suggest_outfit()`
- `new_item` (`dict`) — the listing dict for the thrifted item

**Output:** `str` — a 2–4 sentence social-media caption. If `outfit` is empty or whitespace-only, returns a descriptive error string without calling the LLM; never raises.

---

### `compare_price`

**Purpose:** Estimates whether a listing's price is fair by comparing it against comparable listings in the dataset. Comparables are first narrowed by category and overlapping style tags; if fewer than three are found, the pool falls back to same category only. The item itself is always excluded from the comparison.

**Inputs:**
- `item` (`dict`) — the listing dict to evaluate; must include at least `price`, `category`, and `style_tags`

**Output:** `str` — a human-readable price assessment including a verdict (great deal / fair price / potentially overpriced), the size of the comparison pool, and summary statistics (average, median, min–max range). If the item has no `price` field or no comparable listings exist in the dataset, returns a descriptive message without raising.

---

### `load_style_profile`

**Purpose:** Loads a persisted style profile for a given user from `profiles/{user_id}.json`. The profile encodes preferences learned across past sessions (preferred colors, style tags, size, budget, wardrobe summary) so the agent can pre-fill query defaults and enrich outfit prompts without asking the user to re-describe their wardrobe.

**Inputs:**
- `user_id` (`str`) — a stable identifier for the user (e.g., username or UUID)

**Output:** `dict` — a profile dict with keys `preferred_colors` (`list[str]`), `preferred_styles` (`list[str]`), `preferred_size` (`str | None`), `max_budget` (`float | None`), `preferred_brands` (`list[str]`), `wardrobe_summary` (`str`), and `session_count` (`int`). If the file does not exist or is malformed, returns an empty default profile — never raises.

---

### `save_style_profile`

**Purpose:** Persists an updated style profile to `profiles/{user_id}.json` after a completed session. Merges new signals extracted from the session — the selected item's colors, style tags, size, and price — into the existing profile dict, increments `session_count`, and writes the result to disk. This is how the agent accumulates style memory over time.

**Inputs:**
- `user_id` (`str`) — the same identifier passed to `load_style_profile`
- `session` (`dict`) — the completed session dict from `run_agent()`; must contain at least `selected_item`, `parsed`, and `profile`

**Output:** `str` — a short confirmation string (e.g., `"Style profile updated (session 4)."`) on success, or a descriptive error string if the write fails. Never raises — profile persistence failure does not abort the session.

---

## Planning Loop

At the start of every session the agent calls `load_style_profile(user_id)` to retrieve the user's saved preferences. It then parses the natural language query with regex to extract a `description`, optional `size`, and optional `max_price`, falling back to the profile's `preferred_size` and `max_budget` for any field the user did not specify. The merged parameters are passed to `search_listings`. An empty result triggers an early exit. Otherwise the agent calls `compare_price` on the top listing, then `suggest_outfit` (using the profile's `wardrobe_summary` as supplementary context when the passed-in wardrobe is empty), then `create_fit_card`. An invalid outfit string triggers an early exit. On success the agent calls `save_style_profile(user_id, session)` to merge the selected item's signals into the stored profile before returning. The loop terminates by returning the completed session dict, which the caller inspects via `session["fit_card"]` (success) or `session["error"]` (early exit).

---

## State Management

All state for a single interaction is stored in one in-memory `session` dict initialized by `_new_session()` at the start of `run_agent()`. The dict holds the original query, parsed parameters, search results, selected item, wardrobe, outfit suggestion, fit card, error field, and a `profile` key containing the dict returned by `load_style_profile`. Each tool's output is written into the appropriate key and immediately read by the next step as a direct argument, with no copies made. Cross-session persistence lives exclusively in `profiles/{user_id}.json` — written by `save_style_profile` at the end of each successful session and read back by `load_style_profile` at the start of the next. Within a session everything passes through the dict; once `run_agent()` returns the dict is discarded.

## Error Handling

| Tool | Failure mode | Agent response |
|------|--------------|----------------|
| `search_listings` | No listings match the description, price, or size filters | Returns `[]`; `run_agent` sets `session["error"]` and exits early without calling the remaining tools |
| `suggest_outfit` | Wardrobe is empty (`wardrobe["items"] == []`) | LLM is prompted for general styling advice; the profile's `wardrobe_summary` is appended to the prompt when available |
| `create_fit_card` | `outfit` is an empty or whitespace-only string | Returns the string `"Error: outfit suggestion is missing or incomplete — cannot generate a fit card."` without ever calling the LLM |
| `load_style_profile` | File missing or malformed JSON | Returns an empty default profile; session continues as a first-time user |
| `save_style_profile` | Directory missing or write permission denied | Returns a descriptive error string; session result is still returned to the caller — persistence failure does not abort the session |

**Concrete examples from testing:**

- `search_listings("designer ballgown size XXS under $5")` against the real dataset returned `[]`, and the agent's no-results branch fired correctly — `session["error"]` was set, `session["fit_card"]` stayed `None`, and `suggest_outfit` was never called (verified with a spy in `agent.py`'s `__main__` block).
- `create_fit_card("", SAMPLE_ITEM)` returned an error string without touching `_get_groq_client` — confirmed by `test_empty_outfit_does_not_call_llm` in `tests/test_tools.py`, which asserts the mock was never instantiated.
- `suggest_outfit(SAMPLE_ITEM, {"items": []})` sent a prompt containing `"wardrobe is empty"` to the LLM and returned its response — confirmed by `test_empty_wardrobe_prompt_describes_general_styling`.

## AI Usage

I gave Claude the specs for `search_listings` and `suggest_outfit` — inputs, outputs, and expected behavior — and asked it to generate the initial implementations. The code it produced ran, but returned off outputs: `search_listings` was over-filtering results and `suggest_outfit` was generating vague suggestions that didn't reference actual wardrobe pieces. I then asked Claude to revise both functions based on the specific failures I observed, and also had it add guardrails — input validation to reject malformed or missing fields before any logic ran, and output checks to ensure neither function could silently return an empty or nonsensical result to the caller.

---

## Spec Reflection

The implementation matched the planning spec closely: the three-tool sequence, state-via-session-dict approach, and all three error branches behaved exactly as designed. One minor divergence was that `suggest_outfit` never silently returns an empty string — the LLM always produces output — so the only real failure mode in practice is a missing API key, which raises a `ValueError` before any network call is made. If I were extending the project, I would add a tool that lets the user refine their search interactively (e.g., "show me more options") rather than always picking the top result automatically, since several test runs surfaced interesting second-ranked listings that were a better fit for the stated vibe.
